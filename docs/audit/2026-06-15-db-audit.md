# Audit DB bibs — ottimizzazioni schema/query + squash migrazioni (2026-06-15)

## 1. Sintesi esecutiva

- **Stato del DB**: schema solido (PostGIS + pg_trgm + tsvector, convenzioni `text({enum})`+CHECK, soft-delete, CAS su stock/order-status), ma con alcune lacune sistematiche concentrate su integrità referenziale ai bordi del ciclo di vita account, idempotenza webhook, e determinismo della paginazione.
- **Ottimizzazioni confermate per severità**: **1 P0** (idempotenza webhook Stripe), **6 P1**, **9 P2**, **10 P3** (alcune con severità *adjusted* rispetto alla proposta iniziale — vedi note per-finding). Le ottimizzazioni sono tutte verificate su codice reale (file:line) e nessuna è nuova rispetto agli input.
- **Una sola P0 è bloccante a livello di prodotto**: il dedup webhook viene scritto *prima* dell'handler e mai rollbackato in caso di fallimento → un evento fallito (es. checkout pagato che non provisiona lo store) resta **permanentemente perso**, senza retry Stripe né replay manuale. Da fixare per primo, è logica pura (nessuna migrazione).
- **Temi ricorrenti**: (a) offset-pagination senza `ORDER BY` (3 finding, stessa classe di bug); (b) FK con `cascade`/`set null`/`restrict` incoerenti sul percorso eliminazione account (3 finding di integrità); (c) colonne enumerate senza CHECK a livello DB (P1 su `orders.type/status`); (d) read-check-then-write senza CAS su risorse condivise (punti loyalty, Stripe customer).
- **Squash migrazioni**: **raccomandato e sicuro.** Progetto dev-only mai deployato, ogni dev ricostruisce con `bun run db:reset`; un `drizzle-kit generate` from-scratch riproduce la stessa DDL finale tranne due classi di statement non-rigenerabili (1 `CREATE EXTENSION pg_trgm` + 2 backfill dati), entrambe dimostrabilmente innocue su DB fresco.

## 2. Compattazione migrazioni (squash)

**Verdetto: squash raccomandato.** Le 18 migrazioni (0000–0017) si comprimono in un singolo `0000_init` rigenerato dallo schema corrente. Razionale: il progetto non è deployato, non esiste storico migrazioni né dati in-place da preservare (ogni dev fa `db:reset` = `compose down -v` → `up` → `db:migrate` → `db:seed`), e un `generate` from-scratch riproduce la DDL finale identica eccetto due classi di statement che non si rigenerano — entrambe provatamente moot su volume vuoto. Nessun trigger/funzione/view/colonna generata/`COMMENT ON`/raw SQL esiste in alcuna migrazione (verificato via grep 0000–0017); PostGIS è creato dall'initdb Docker, non da migrazioni, ed è correttamente escluso dai diff via `extensionsFilters:["postgis"]`.

### Cosa NON si rigenera da `drizzle-kit generate`

| Item | Da quale migrazione | Come gestirlo |
|---|---|---|
| `CREATE EXTENSION IF NOT EXISTS pg_trgm;` | `0006_sharp_vulture.sql` (riga 1) | **NON riemesso** da drizzle-kit per gli indici `gin_trgm_ops` (`extensionsFilters` lista solo postgis). **Prependere manualmente** come primo statement del `0000` rigenerato. Ridondante su DB dev (lo crea `docker/postgis/initdb-postgis.sh:24`), ma rende la migrazione self-contained e robusta contro un Postgres non-postgis. |
| **Backfill** `UPDATE order_items SET product_name/product_ean/brand_name/product_image_url/product_id = <snapshot>` | `0004_products_status_order_items_snapshot_audit_log.sql` (righe 24–41) | **DROP — non riprodurre.** Backfill one-time delle colonne snapshot denormalizzate per ordini pre-esistenti. Lo schema finale (`order.ts`) dichiara già `product_name NOT NULL`; su DB fresco la tabella è vuota al migrate e il seed inserisce righe con snapshot popolati → no-op. |
| **Backfill** `UPDATE seller_profiles SET onboarding_status='pending_review' WHERE onboarding_status IN ('pending_store','pending_team','pending_payment')` | `0011_cute_slyde.sql` (righe 1–3) | **DROP — non riprodurre.** Data-fix one-time per un rename enum. I tre status legacy non esistono più nell'enum `onboardingStatuses` (`seller.ts`), mai inseriti su DB fresco. La DDL annessa (colonna `stripe_customer_id` + unique) **è** rigenerabile e viene riprodotta automaticamente. |
| postgis + topology/fuzzystrmatch/tiger + colonne `geometry(point)` + indici gist | Nessuna migrazione — provisioned da `docker/postgis/initdb-postgis.sh` | **NESSUNA AZIONE.** postgis escluso dai diff via `extensionsFilters:["postgis"]`; le colonne geometry (`customer_addresses.location`, `stores.location`) e i loro indici gist **sono** rigenerati normalmente. L'estensione resta fornita dall'immagine Docker (garantita da `db:reset`). |

### Procedura (copia-incollabile)

```bash
# 1. Stato git pulito + feature branch (regola repo: mai commit su main)
git checkout -b chore/squash-migrations

# 2. Baseline DDL canonica dalle migrazioni VECCHIE
bun run db:reset
docker exec bibs-postgis pg_dump -U pgadmin -d bibs-db \
  --schema-only --no-owner --no-privileges > /tmp/ddl_before.sql

# 3. Elimina l'intera cartella migrations (incl. meta/_journal.json) → niente hash-bootstrap
rm -rf /Users/marcogelli/repos/jelaz/bibs/apps/api/src/db/migrations
#    equivalente: bun run --cwd apps/api db:clean

# 4. Rigenera un singolo baseline dallo schema corrente
bun run --cwd apps/api db:generate
```

**Edit manuale** — aprire il nuovo `apps/api/src/db/migrations/0000_*.sql` e aggiungere come **primissimo** statement, prima del primo `CREATE TABLE`:

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;--> statement-breakpoint
```

(NON aggiungere `CREATE EXTENSION postgis` — è di competenza dell'initdb Docker ed è filtrato dai diff.)

```bash
# 5. Sanity-check del 0000 rigenerato
grep -c 'CREATE TABLE'  apps/api/src/db/migrations/0000_*.sql   # = n. tabelle schema live
grep gin_trgm_ops       apps/api/src/db/migrations/0000_*.sql   # attesi brands_name_trgm_idx + product_name_trgm_idx
grep -niE '^\s*(update|insert|delete)' apps/api/src/db/migrations/0000_*.sql  # deve essere VUOTO (no backfill)

# 6. Rebuild pulito completo sul nuovo set
bun run db:reset

# 7. Diff DDL post-squash vs baseline (solo differenze cosmetiche ammesse)
docker exec bibs-postgis pg_dump -U pgadmin -d bibs-db \
  --schema-only --no-owner --no-privileges > /tmp/ddl_after.sql
diff /tmp/ddl_before.sql /tmp/ddl_after.sql

# 8. Test (testcontainers applica la cartella migrazioni)
bun run --cwd apps/api test:integration
bun run --cwd apps/api test:unit
bun run --cwd apps/api typecheck
```

Se verde: committare **solo** la cartella migrations (nuovo 0000 + meta) e aprire PR. Nel body avvisare che è uno squash dev-only che **richiede a ogni dev `bun run db:reset` dopo il pull** (ledger migrazioni resettato).

### Verifiche post-squash

- `bun run db:reset` esce con codice 0 (down -v + up --wait + migrate + seed verdi su volume vuoto).
- `diff /tmp/ddl_before.sql /tmp/ddl_after.sql` mostra **zero** differenze strutturali — ogni tabella, colonna (incl. NOT NULL/default), FK, CHECK, unique/partial index, gist index, tsvector gin index e i due `gin_trgm_ops` index sono identici pre/post.
- `SELECT extname FROM pg_extension;` include sia `postgis` sia `pg_trgm`.
- Trigram search funziona: `EXPLAIN` su una query `lower(name) % 'foo'` usa i `*_trgm_idx` (o almeno non erra con "operator does not exist").
- `test:integration`, `test:unit`, `typecheck` verdi.
- grep conferma: zero `UPDATE/INSERT/DELETE` nel 0000 (backfill spariti) ed esattamente un `CREATE EXTENSION IF NOT EXISTS pg_trgm;` (postgis NON ri-aggiunto a mano).

**Rischi noti**: (1) i nomi/predicati di index/constraint seguono lo *schema corrente*, non lo storico (es. `product_seller_ean_unique` ha `WHERE ean IS NOT NULL AND status != 'trashed'`) → validare via diff `pg_dump`, non leggendo il SQL. (2) Eventuali dev con DB locale long-lived da non resettare sono forzati a `db:reset` (perdita dati locali) — accettabile per il workflow, ma da segnalare prima del merge.

## 4. Piano d'azione consigliato

Ordine raccomandato: prima il **P0 webhook** (rischio prodotto concreto, logica pura), poi lo **squash + indici/cleanup index** (lavoro infrastrutturale che semplifica tutto il resto), poi **integrità/concorrenza** (FK + CAS + CHECK), infine **data-model** hardening. Raggruppato in 3 PR logiche:

### PR 1 — `fix(webhooks): atomic stripe event dedup` (P0, urgente, isolata)
- Solo il fix dispatcher Stripe (claim atomico su `processed_at`, delete su fallimento). Nessuna migrazione. Test deterministico idempotency.
- **Va in testa, indipendente da tutto il resto.**

### PR 2 — `chore(db): squash migrations + index cleanup` (P2/P3 indici + squash)
1. **Squash** (sezione 2) — singolo `0000_init`, è il prerequisito che assorbe i `DROP INDEX` successivi in un baseline pulito invece di accumulare migrazioni.
2. Eseguire **dentro lo stesso branch, prima del generate finale**, le modifiche schema a impatto-indice così che il `0000` rigenerato le incorpori direttamente:
   - `+ store_product_store_id_idx`, `+ store_category_id_idx` (P2 indici mancanti);
   - `- order_status_idx`, `- order_type_status_idx` (P3 ridondanti);
   - `orderBy` su `listMunicipalities` e `listAddresses` (vedi sotto, sono codice non-schema ma raggruppabili qui se si preferisce — altrimenti in PR3).
- Verifiche: diff `pg_dump` before/after deve mostrare i due nuovi indici e l'assenza dei due rimossi; `test:integration` verde.

### PR 3 — `fix(db): referential integrity, CAS guards & enum checks` (P1/P2/P3 integrità + concorrenza + data-model)
- **Integrità (P1)**: CHECK su `orders.type/status` (+ le altre enum mancanti); `orders.customer_profile_id` → `set null` nullable.
- **Integrità (P3)**: `stores.seller_profile_id` → `restrict`; `point_transactions.order_id` → `cascade`; `order_items.vat_rate` CHECK; (opzionale) `session.impersonated_by` FK.
- **Concorrenza (P1/P2/P3)**: CAS punti loyalty (`orders.ts` + 409); CAS verify/rejectSeller; CAS Stripe customer; estensione partial-unique `redeemed`; transazione/catch in `deleteProductMacroCategory`.
- **Data-model (P2/P3)**: `user.birth_date` → `date`; Better Auth timestamp → `timestamptz`; `order_items.vat_rate/vat_amount` → NOT NULL (+ edit dei 2 insert di test in `seller-orders.test.ts`).
- **N+1/query (P2/P3)**: batch upsert brand CSV (con dedup case-insensitive); `imageOrder` set-based; `columns:{id:true}` su ownership guards; filtro `pending` su `listEmployeeInvitations`.
- Differire (no fix ora): over-fetch order-list `storeProduct.product` (P3/P4) — riapplicare quando arriva la UI ordini con uno schema list-only dedicato.

Tutte le modifiche schema generano migrazioni via `db:generate` (mai hand-write, mai `db:push` — deny-listed) e, post-squash, partono da un singolo baseline pulito. Ogni PR chiude con `test:integration` + `test:unit` + `typecheck` verdi.
